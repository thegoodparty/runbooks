# Campaign Event Generation

Reference for the AI-powered community event generation system that finds local events for political candidates.

## Prerequisites

**books/.env variables**: `$AWS_REGION`, `$AWS_PROFILE`

## Overview

Candidates get community event tasks (town halls, festivals, parades, etc.) generated by an AWS Lambda that uses Gemini 3 Flash with Google Search grounding. The Lambda is triggered by gp-api via SQS when a new user opens their campaign dashboard for the first time.

## Architecture

```
gp-webapp                  gp-api                              AWS                              gp-api (consumer)
────────                   ──────                              ───                              ─────────────────
GET /tasks/generate/stream
                           1. Create default tasks
                           2. Check nonDefaultTasksExist
                           3. Send SQS message ──────────────► campaign-plan-input-{env}.fifo
                           4. Poll DB for non-default tasks                │
                                                                           ▼
                                                               campaign-plan-{env} Lambda
                                                               ├─ Gemini + Google Search grounding
                                                               ├─ Find 5-8 community events
                                                               ├─ Write result JSON to S3
                                                               └─ Send CAMPAIGN_PLAN_COMPLETE ──► {env}-Queue.fifo
                                                                                                       │
                           5. Poll sees non-default tasks                                         5. Read S3 result
  ← complete: [all tasks]  6. Return all tasks                                                    6. Save event tasks
                                                                                                  7. Notify Slack
```

## Lambda Input/Output

### Input (SQS message body)

```json
{
  "campaignId": 12345,
  "election_date": "2026-11-04",
  "city": "Boston",
  "state": "MA"
}
```

Requires `city` and `state` from `campaign.details`. ~80% of pro campaigns don't have `city` populated — those campaigns skip event generation and get only default tasks.

### Output (S3 JSON)

```json
{
  "campaignId": 12345,
  "tasks": [
    {
      "title": "Boston Pride Festival and Parade",
      "description": "Large celebration offering engagement with diverse voters.",
      "cta": "Attend event",
      "flowType": "events",
      "week": 22,
      "date": "2026-06-06",
      "url": "https://www.bostonprideforthepeople.org"
    }
  ],
  "taskCount": 7,
  "generationTimestamp": "2026-04-02T15:30:00.000000+00:00"
}
```

The `url` field is optional — validated to require `http://` or `https://` scheme.

### Completion Message (SQS to gp-api)

**Success:**
```json
{
  "type": "campaignPlanComplete",
  "data": {
    "campaignId": 12345,
    "status": "completed",
    "s3Key": "results/12345/2026-04-02T15:30:00-abc12345.json",
    "taskCount": 7,
    "generationTimestamp": "2026-04-02T15:30:00.000000+00:00"
  }
}
```

**Error:**
```json
{
  "type": "campaignPlanComplete",
  "data": {
    "campaignId": 12345,
    "status": "error",
    "error": "Campaign plan generation failed"
  }
}
```

## Prompt Management

Prompts are managed in Braintrust (project: "campaign-plan") with hardcoded fallbacks in `campaign_plan_lambda/event_generator.py`. Two prompts:

| Slug | Purpose |
|------|---------|
| `search-community-events` | Google Search grounding query for finding events |
| `filter-and-structure-events` | Filter, rank, and structure events from search results |

## Key Design Decisions

- **No CampaignPlan table dependency** — event tasks are saved directly to `campaign_task`. CampaignPlan is not written to or read from in the new flow.
- **Append-only tasks** — `addTasks` uses `createMany` with `skipDuplicates`. Tasks are never deleted by the event generation flow.
- **Idempotent processing** — task IDs are deterministic (`event-{campaignId}-{index}-{generationTimestamp}`). Duplicate SQS delivery produces no duplicates.
- **Content-based SQS deduplication** — `MessageDeduplicationId` is a SHA256 hash of the message body. Same campaign + same inputs within 5 minutes = deduplicated.
- **Prod disabled** — `CAMPAIGN_PLAN_COMPLETE` handler has a `NODE_ENV === 'production'` guard. Queue URL and S3 bucket are empty in prod Pulumi config.
- **Zod validation at boundaries** — `CampaignPlanCompleteMessageSchema` (discriminated union) validates incoming SQS messages. `LambdaResultPayloadSchema` validates S3 payloads.

## Local Development

### Full end-to-end (gp-api + Lambda locally)

1. Start the local Lambda server in gp-ai-projects:
```bash
cd gp-ai-projects
source .venv/bin/activate
python campaign_plan_lambda/local_server.py
```

2. Set in gp-api `.env`:
```
CAMPAIGN_PLAN_LOCAL_URL=http://localhost:8089/generate
CAMPAIGN_PLAN_RESULTS_BUCKET=campaign-plan-results-dev
```

3. The local server needs in gp-ai-projects `.env`:
```
GEMINI_API_KEY=...
AWS_PROFILE=your-admin-profile
AWS_DEFAULT_REGION=us-west-2
S3_RESULTS_BUCKET=campaign-plan-results-dev
OUTPUT_SQS_QUEUE_URL=https://sqs.us-west-2.amazonaws.com/333022194791/YourName_Queue.fifo
```

Each engineer should have their own dedicated SQS queue (e.g., `YourName_Queue.fifo`).

### Testing without Lambda (prod-like behavior)

Leave `CAMPAIGN_PLAN_LOCAL_URL` and `CAMPAIGN_PLAN_INPUT_QUEUE_URL` empty. The SSE stream will create default tasks and skip event generation.

## Troubleshooting

### Events not generating

1. Check campaign has `city` and `state` in details: `SELECT details->>'city', details->>'state' FROM campaign WHERE id = {id}`
2. Check `CAMPAIGN_PLAN_INPUT_QUEUE_URL` or `CAMPAIGN_PLAN_LOCAL_URL` is set
3. Check Lambda CloudWatch logs: `aws logs tail /aws/lambda/campaign-plan-{env} --follow`

### Duplicate tasks

Tasks use deterministic IDs and `skipDuplicates`. If duplicates appear, check:
1. Are task IDs being passed through `mapTasksToCreateData`? (the `...(task.id && { id: task.id })` spread)
2. Is the Lambda sending different `generationTimestamp` for the same generation?

### Lambda errors

- **Blocked response** — Gemini refused the query. Not retried (goes to DLQ after max retries).
- **Invalid dates** — all LLM-returned events had unparseable dates. Retried up to 3 times.
- **Empty results** — no events found for the area. Returns empty task list (success).

### SQS message stuck

Check the input queue: `aws sqs get-queue-attributes --queue-url {queue-url} --attribute-names All`

Messages in flight = being processed. Messages visible = waiting. DLQ messages = failed 3 times.

## Monitoring

- **DLQ alarm** → SNS → Slack notifier. Fires when any message lands in the dead letter queue.
- **Slack notification** — sent on successful event generation via `notifySlackCampaignPlanCreated` (channel: `casClickupTasks`).
- **Braintrust** — LLM call tracing at `braintrust.com` (project: "campaign-plan").
